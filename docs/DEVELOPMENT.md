# Development Guide

## Architecture

4-layer design:

```text
Layer 1: Safe Rust wrappers    (src/*.rs)
Layer 2: Rust FFI declarations (ocio-sys/src/lib.rs)       [AUTO-GENERATED]
Layer 3: C++ bridge             (ocio-sys/src/bridge.cpp)  [AUTO-GENERATED]
Layer 4: OCIO native library    (libOpenColorIO.a)
```

Layers 2 and 3 are generated by `tools/generator/` from OCIO C++ headers.

## Build System

### Stub mode (default)

```bash
cargo build
cargo test
```

`OCIO_RS_STUB` is defined; bridge.cpp compiles without OCIO headers. All C++ functions return safe defaults (nullptr, 0, false).

### Real OCIO mode

```bash
# Bundled: cmake builds OCIO from the submodule
cargo build --features bundled

# Pre-installed OCIO
OCIO_RS_ENABLE_REAL=1 OCIO_INSTALL_DIR=/path/to/ocio cargo build

# OCIO source tree
OCIO_RS_ENABLE_REAL=1 OCIO_SOURCE_DIR=/path/to/ocio cargo build
```

Static linking (Windows) requires: OpenColorIO, expat, yaml-cpp, Imath, pystring, minizip-ng, zlib, plus system libs (gdi32, user32, etc.). With `--features bundled`, OCIO's cmake fetches and builds all transitive dependencies automatically.

## Code Generator

`tools/generator/` is a standalone Rust project (outside the Cargo workspace).

```bash
cd tools/generator
cargo run                             # Generate all files
cargo run -- --check                  # Dry-run: parse only
cargo run -- --ocio-path <path>       # Use custom OCIO headers
```

Generated files:

- `ocio-sys/src/bridge.hpp` — C FFI declarations
- `ocio-sys/src/bridge.cpp` — Stub/real dual-mode implementation
- `ocio-sys/src/lib.rs` — Rust FFI declarations

Design decisions:

- Regex-based parsing of OCIO C++ headers (no full C++ parser needed)
- `#ifdef OCIO_RS_STUB` for dual-mode code
- Overloaded methods disambiguated with `_v2`, `_v3` suffixes
- `std::shared_ptr<void>` for type-erased handle storage
- C++ exceptions mapped to error returns (nullptr, 0, false)

Known issues: the generator's type-mapping still produces a number of edge-case errors in real-OCIO mode. `bridge.cpp` is manually patched after regeneration. See the commit history for the patch patterns.

## OCIO Version Strategy

`third_party/OpenColorIO` is a git submodule pinned to a specific OCIO release. ocio-rs uses its own semver tracking the OCIO series it targets: `0.1.x` maps to OCIO `v2.5.x`.

Upgrade workflow: update submodule → run generator → fix compile errors → release.

## Testing

```bash
# Stub mode (default)
cargo test --workspace

# Real OCIO mode
cargo test --features bundled

# API parity checker (L1: FFI coverage, L2: wrapper coverage, L3: API diffs)
cargo run --bin check_parity
cargo run --bin check_parity -- --check-l3

# Benchmarks
cargo bench
cargo run --release --bin bench_ocio_rs
```

## Performance

FFI overhead: ~5-8 ns per pixel call. Use `apply_rgba_pixels()` for batch processing — it amortizes the FFI cost across thousands of pixels per call.

Reference (MatrixTransform + applyRGBA, 1M iterations):

- C++ direct: ~7-10 ns/pixel
- Rust FFI: ~15 ns/pixel

## Continuous Integration

Current CI (`ci.yml`): `cargo check --workspace --no-default-features` (stub mode, no OCIO required).

Recommended full CI pipeline:

```yaml
steps:
  - cargo test --workspace
  - cargo run --bin check_parity
  - cargo clippy -- -D warnings
  - cargo fmt --check
```

Real-OCIO tests require a built OCIO library, which adds significant build time. Stub-mode tests provide sufficient coverage for most PRs.
