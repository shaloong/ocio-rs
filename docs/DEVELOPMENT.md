# Development Guide

## Architecture Overview

ocio-rs uses a 4-layer architecture:

```text
Layer 1: Safe Rust wrappers    (src/*.rs)
Layer 2: Rust FFI declarations (ocio-sys/src/lib.rs)       [AUTO-GENERATED]
Layer 3: C++ bridge             (ocio-sys/src/bridge.cpp)  [AUTO-GENERATED]
Layer 4: OCIO native library    (libOpenColorIO.a)
```

Layers 2 and 3 are generated by `tools/generator/` from OCIO C++ headers.

## Build System

### Stub Mode (default)

```bash
cargo build          # No OCIO dependency needed
cargo test           # All tests pass with stub implementations
```

Stub mode defines `OCIO_RS_STUB` which causes bridge.cpp to compile without OCIO headers. All functions return safe defaults (null, zero, false).

### Real OCIO Mode

```bash
OCIO_RS_ENABLE_REAL=1 OCIO_SOURCE_DIR=third_party/OpenColorIO cargo build
```

Static linking dependencies (Windows): OpenColorIO, expat, yaml-cpp, Imath-3_2, pystring, minizip-ng, zlib, bz2, lzma, zstd, bcrypt, gdi32, and other system libs.

## Code Generator

The code generator at `tools/generator/` is a standalone Rust project (outside the Cargo workspace) that:

1. Parses OCIO C++ headers from `third_party/OpenColorIO/include/OpenColorIO/`
2. Extracts class/method definitions using regex
3. Generates three files:
   - `ocio-sys/src/bridge.hpp` — C FFI function declarations
   - `ocio-sys/src/bridge.cpp` — Stub/real dual-mode implementations
   - `ocio-sys/src/lib.rs` — Rust FFI declarations

### Usage

```bash
cd tools/generator
cargo run -- --check                  # Dry-run, parse only
cargo run                             # Generate all files
cargo run -- --ocio-path <path>       # Use custom OCIO headers
```

### Key Design Decisions

- Written in Rust (no Python dependency)
- Regex-based parsing (not full C++ parser — OCIO headers follow consistent patterns)
- Generates stub/real dual-mode code via `#ifdef OCIO_RS_STUB`
- Handles overloaded methods with `_v2`, `_v3` naming suffixes
- Uses `std::shared_ptr<void>` for type-erased handle storage
- Maps C++ exceptions to error returns (nullptr, zero, false)

### Remaining Known Issues

29 type-mapping edge cases in real-OCIO mode:

- `const T&` output parameter dereferencing
- `std::ostream&` serialization parameter
- Const-to-non-const `shared_ptr` conversions for skipped classes
- Missing enum type entries for some OCIO v2.6 types

These are tracked as individual mapping fixes — no architectural changes needed.

## OCIO Version Strategy

1. `third_party/OpenColorIO` is a git submodule pinned to a specific OCIO version
2. When OCIO releases a new version: update submodule → run generator → fix any compile errors → release
3. ocio-rs version numbers mirror OCIO versions (e.g., ocio-rs v2.4.x for OCIO v2.4.x)

## Testing

```bash
# All tests
cargo test --workspace

# Integration tests
cargo test --test matrix_op_tests

# API parity check
cargo run --bin check_parity
cargo run --bin check_parity -- --check-l3  # Requires OCIO source

# Benchmarks
cargo bench
cargo run --release --bin bench_ocio_rs
```

## Performance Notes

The FFI overhead is ~5-8 ns per pixel call. For batch processing, use `apply_rgba_pixels()` which amortizes this cost across thousands of pixels per FFI call.

Benchmark reference (MatrixTransform + applyRGBA, 1M iterations):

- C++ direct: ~7-10 ns/pixel
- Rust FFI: ~15 ns/pixel

## Continuous Integration

Recommended CI steps:

```yaml
steps:
  - cargo test --workspace
  - cargo run --bin check_parity
  - cargo clippy -- -D warnings
  - cargo fmt --check
```

Note: Real-OCIO tests require a built OCIO library, which adds significant CI time. Stub mode tests provide sufficient coverage for most PRs.
